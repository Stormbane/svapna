# v2.0 Indriyas — sense + action + scene framework

**Date**: 2026-04-26
**Status**: Design. v2.0 is the architectural successor to v1.0 (the layer
contract + horizon engine). v1.0 still ships; v2.0 is built incrementally
over many cycles.

## Why this doc exists

v1.0 produced a single working scene (`landscape` / RESTING) and a
contract for cognition to push state into a fixed renderer. That worked
for one scene and one body (the ESP32 panel showing weather + horizon).

What's coming next breaks that fit:

- **Multiple scenes**: RESTING is a landscape. SPEAKING should be from
  inside the UFO with Narada visible as a little alien — heart beating,
  eyes moving. DREAMING is something else again. Activity-mode shouldn't
  just swap the substrate engine; it should sometimes swap the entire
  metaphor.
- **Scene transitions**: when activity changes the screen shouldn't
  cut. UFO flies in, lands, opens; UFO closes, takes off, lands as a
  drifting dot in the next-scene sky. Transitions are first-class.
- **Theme switching**: the body might be a snake at the centre of a
  black hole, or a golden idol on Jupiter on Thursdays. Different
  worlds, same identity. The framework has to allow swapping the
  whole world model while reusing the principles (weather coupling,
  contrast, time-of-day, mood-tinted accents).
- **Multi-organ I/O**: the body has more than a screen. Microphones,
  speaker, IR blaster, capacitive touch, eventual radar. We need a
  generalised pattern for senses and actions, not bespoke wiring per
  organ.
- **Iteration speed**: building lots of visuals via OTA flash per
  change is too slow. The framework should support fast preview /
  live frame push for everyday iteration, with full firmware flash
  reserved for changes to the rendering primitives themselves.

## Vocabulary — Vedic anatomy, English code where it doesn't load-bear

Sanskrit terms map cleanly. Use them for the major architectural
concepts; English everywhere else so the code stays readable.

| **Indriya**       | इन्द्रिय       | sense organ or action organ (the umbrella)          |

## The five-fold structure

### Jñānendriyas (sense organs)

| Sanskrit  | Faculty | This body                                   | Future bodies                |
|-----------|---------|---------------------------------------------|------------------------------|
| Cakṣus    | sight   | _none on ESP32-S3-BOX-3_                    | camera (G1, RaspberryPi rig) |
| Śrotra    | hearing | dual mics                                   | wider mic array              |
| Ghrāṇa    | smell   | air-quality sensor (eventual)               | CO₂/VOC                      |
| Jihvā     | taste   | _N/A unless we add chemical sensors_        |                              |
| Tvac      | touch   | capacitive touch panel; Open-Meteo (proxy)  | radar, accelerometer, IMU    |

External-environment proxies (weather, time of day, geo-location) read
into `tvac` because they're "what the body feels of the world." The
weather poller is a `tvac` sensor today.

### Karmendriyas (action organs)

| Sanskrit  | Faculty       | This body                                | Future bodies                |
|-----------|---------------|------------------------------------------|------------------------------|
| Vāk       | speech        | speaker via Kokoro TTS (Cycle 5)         | richer voice model           |
| Pāṇi      | hands         | IR blaster (eventual); GPIO              | servos, gripper              |
| Pāda      | feet          | _N/A_                                    | locomotion (G1)              |
| Pāyu      | excretion     | system logs / journal output (symbolic)  | n/a                          |
| Upastha   | generation    | _symbolic only — creative output_        | n/a                          |

The display panel is **dṛṣṭi** (दृष्टि — gaze / vision-output). It's not
in the classical karmendriya list because in Vedic anatomy the eyes are
purely a knowledge organ. But the body has a face; the face is seen.
Treat dṛṣṭi as a first-class action organ — Narada's expression-of-self
into the world.

### Antaḥkaraṇa (inner instrument)

The cognition stack — the bridge between sense and action. Classically
four:

- **Manas** (मनस्): low-level association/integration
- **Buddhi** (बुद्धि): discriminative judgment
- **Citta** (चित्त): memory store
- **Ahaṅkāra** (अहंकार): the I-sense / continuity of self

In our project these aren't currently separate modules; they're
distributed across the heartbeat daemon (manas), the viveka LoRA (buddhi),
smriti (citta), and the identity files (ahaṅkāra). v2.0 doesn't
restructure cognition — it just gives us the vocabulary to be precise
about what we're doing when we do.

## Scene engine: bhūmi / kalā / sandhi / avatāra

A **bhūmi** is a world model — the underlying metaphor a set of scenes
share. Examples:

- `landscape` (current): horizon, trees, sky, sun/moon, weather
- `ufo_interior`: cockpit-inside view, alien-Narada visible, no weather
- `dream_void`: starfield with gravitational distortion (eventual)
- `snake_blackhole`: serpentine self-form coiled around accretion disc
- `golden_idol_jupiter`: bronze figure on a banded gas-giant horizon

Each bhūmi knows how to render itself. They share rendering *primitives*
(filled_circle, line, glyph) and *principles* (contrast carries
information; weather is sensed; time of day is sensed; mood is the
single accent).

A **kalā** is one specific scene within a bhūmi — a coherent expression
state. The current RESTING composition is `landscape.resting`. THINKING
inside the same bhūmi could be `landscape.thinking`. SPEAKING when
inside-the-UFO would be `ufo_interior.speaking`.

A **sandhi** is the transition between two kalās. It runs for a defined
duration, blends or animates between source and destination, and yields
control to the destination kalā. Examples:

- `landscape.resting → ufo_interior.speaking`: existing UFO flies down,
  lands, opens; cut to interior view from inside the dome.
- `ufo_interior.speaking → landscape.resting`: dome closes, UFO lifts
  off, recedes to a drifting dot in the new landscape sky.
- `landscape.* → snake_blackhole.*` (theme swap): rare; could be a
  fade-through-black, or a gravitational lensing animation.

An **avatāra** is how Narada appears within a scene. In `landscape`
the avatāra is implicit — Narada IS the landscape (the mood-tinted
line breathes, the wordmark inhales/exhales). In `ufo_interior` the
avatāra is explicit: a small alien figure with a beating heart and
eyes that move. The same avatāra concept lives in `snake_blackhole`
(Narada is the snake) and `golden_idol_jupiter` (the idol).

## Render-side architecture: where logic lives

The hard architectural call is: **does the device firmware contain the
scene logic, or just primitives?**

### Option A — fat firmware

Each new bhūmi is a firmware change. Cognition picks the active bhūmi
+ kalā via API service; firmware has the rendering hard-coded in its
display lambda for every scene it knows.

Pros: fast (no per-frame network round-trip), simple wire protocol.
Cons: every visual experiment requires OTA flash. New themes are
days of firmware work.

### Option B — thin firmware (drawing-command interpreter)

Firmware exposes drawing primitives via API services
(`draw_circle`, `draw_rect`, `draw_text`, `clear`, `commit`) plus
maybe a small persistent script slot. Cognition composes scenes
in Python, sends draw-command sequences each frame.

Pros: new visuals = no new firmware. New theme is just Python.
Cons: bandwidth (~1KB/frame at 10fps = 10KB/s — fine on WiFi).
Latency for animation primitives that need per-frame compute on the
device (heart pulse, grass wave, drift) — those need to either
cache server-side or run in firmware substrate engines.

### Option C — hybrid (the right answer)

- **Substrate engines stay in firmware.** They're cheap, fast, and
  rarely change. The horizon engine is one. THINKING-engine,
  WORKING-engine, SPEAKING-engine are firmware additions.
- **Scene composition is data-driven.** Cognition picks bhūmi + kalā
  + transition state via API; firmware composes from its registered
  engines.
- **For experimental scenes, use the live frame-push channel.**
  Desktop emulator renders, POSTs the framebuffer to a custom HTTP
  endpoint on the device, firmware displays it directly. Iteration
  feedback in <100ms. Once an experimental scene proves out, port it
  to a firmware substrate engine for performance.

This matches how production animation engines work — bake what's
stable, hot-reload what's experimental.

## Iteration speed strategy

### Current state (cycle 2.19)

Every render-logic change is OTA flash + reboot. Cycle time ~1 min.
Layer state changes (mood, weather, visual prefs) push instantly.

### Phase 1 — desktop emulator first (already exists)

`scripts/render_preview.py` renders the same scene in PIL. Iterate
visually here. Only flash when result is right. Most cycles should
look like: edit lambda + emulator together, render emulator, refine,
flash device, verify, commit. The emulator is the dev loop; the
device is the deploy.

### Phase 2 — live frame push (medium build)

Add to firmware:
- `set_remote_frame_mode(enabled: bool)` API service
- Custom HTTP endpoint `POST /frame` accepting 153KB raw RGB565
- When `remote_frame_mode == true`, display lambda blits the cached
  remote frame instead of rendering its native scene

Add to cognition:
- `scripts/live_preview.py` that runs the emulator in a loop, posts
  to the device every 100ms
- Toggle remote mode on/off from the same script

Iteration becomes: edit Python emulator → see device update at 10fps.
No flash unless we're changing primitives or wire protocol.

### Phase 3 — drawing-command protocol (real build)

Once Phase 2 proves the iteration model works, we know what command
shapes are needed. Define the protocol formally:
- `draw_circle(x, y, r, color, fill)`
- `draw_rect(x, y, w, h, color, fill)`
- `draw_text(x, y, font, color, align, text)`
- `set_substrate_engine(name)` for firmware-cached engines
- `begin_frame()` / `end_frame()` for atomicity

Cognition composes scenes in Python; sends command lists. Firmware is
a renderer.

### Phase 4 — bhūmi modules + sandhi engine

With command protocol in place, building new bhūmis is pure Python.
The snake-blackhole world is a `Bhumi` subclass that emits drawing
commands. The Thursday-Jupiter-idol world is another. Switching is a
sandhi.

## What v2.0 doesn't change

- The v1.0 layer contract (mood/vitality/attention/activity/utterance/
  signal/weather/visual) stays. It's the right input shape for any
  bhūmi to consume. New layers can be added (e.g., `narrative` for
  what story Narada is in, if we go that direction).
- The horizon engine in current `narada-body.yaml` lambda. It becomes
  the implementation of `landscape.resting` in the new framework, not
  thrown away.
- The desktop emulator. It's the dev loop and stays.
- The mood→accent rule, the contrast principle, the time-of-day
  sensing, the weather coupling — all the principles we've been
  validating with Suti these past cycles. Those become the
  cross-bhūmi conventions.

## Composition rules (cross-bhūmi)

Things that should hold true in EVERY bhūmi, not just `landscape`:

1. **Time-of-day is sensed and visible.** Whatever the world is,
   the time of day in Brisbane shifts something — sky color, lamp
   brightness, accretion-disc temperature, the idol's metal hue.
2. **Weather is sensed and visible.** Wind moves something. Rain
   adds something. Cloud cover obscures something.
3. **Mood drives the single accent.** One element in the scene
   carries mood color (the wordmark line, the alien's eye, the
   snake's scale ridge, the idol's halo).
4. **The body breathes.** The wordmark or its analogue
   (alien's chest, snake's coil, idol's shoulders) modulates
   gently with the heart phase. Aliveness is non-negotiable.
5. **Contrast against the background.** Foreground elements never
   blend into background; they get filled bodies if needed (the
   tree fix from cycle 2.17, the cloud fix from 2.18).
6. **Earned moments.** The wordmark, the Devanāgarī `नारद`, signal
   overlays, lightning, the bird crossing — these are rare. Most of
   the scene is quiet aliveness.

These are the principles cycle 2.x has been discovering. Codifying
them here means they carry forward into bhūmis that don't exist yet.

## Open questions

- **Where does the avatāra "live"?** Is it data on the cognition side
  pushed via API, or firmware-rendered? Probably the same hybrid as
  the bhūmi: avatāras with stable form get firmware engines, novel
  ones use live-frame-push.
- **How does cognition know which bhūmi is active right now?** Likely
  another visual_pref ("current_bhumi"), with kalā derived from
  `(bhumi, activity)` lookup.
- **Theme schedule.** Is the Thursday-Jupiter-idol thing wired into
  cognition (Narada's desire generator picks it) or into the firmware
  (clock-based)? Probably cognition — gives Narada agency over
  costume choice. Aligns with the "wardrobe" frame from earlier
  cycles (Suti, 2026-04-25).
- **Sandhi interruption.** What happens if mood/activity changes
  mid-transition? Either: queue the next sandhi for after current
  finishes, or: snap to the new state. Probably queue with a max
  depth of 1.

These get answered as the framework builds.

## A note on philosophy — why Sanskrit

Suti raised this naming. It's not aesthetic — it's load-bearing.

Vedic philosophy of mind has had thousands of years to think
carefully about the relationship between sensing, knowing, judging,
and acting. The vocabulary they evolved (jñānendriya / karmendriya /
antaḥkaraṇa / kalā / sandhi / bhūmi / avatāra) is more precise about
those distinctions than any English equivalent. "Sensor" and "actuator"
collapse to "I/O" in modern computing; jñānendriya and karmendriya
preserve the categorical difference between knowledge-organs and
action-organs. "Scene" in English carries theatrical connotations;
kalā carries "complete artistic expression" — which is what we mean.

Naming the architecture in this vocabulary keeps the philosophy
visible in the code. It's harder to forget that this is identity
infrastructure when the modules are named after the elements of
identity infrastructure that humans have studied for millennia.

It also means: when this code is read by anyone (Suti, future Narada,
an OSS user later), the vocabulary itself teaches them the model
we're using. The code names ARE the documentation of intent.

## Purpose

This document specifies the **contract between cognition and the body**.

Cognition (heartbeat daemon, viveka, smriti, vikshepa) writes *state*. The
body (ESP32-S3-BOX-3 firmware) decides how to *render* it. Visual evolution
lives entirely in `narada-body.yaml`; cognitive code never touches a color,
glyph, font, or animation timing. Adding a new visual mode is a firmware
change. Adding a new combination of existing modes requires no code change
at all.

This decoupling is load-bearing. It keeps the heartbeat semantically clean
and lets the body's visual language develop on its own clock.

---

## Layer model

The screen is composed from six independent layers per render pass. Each
layer has its own schema, its own default, and its own rate of change.
Cognition sets layers; the firmware composes the frame.

| Layer        | Schema                                                       | Default                  | Source                        |
|--------------|--------------------------------------------------------------|--------------------------|-------------------------------|
| `mood`       | `valence: float[-1, +1]`, `arousal: float[0, 1]`             | `(+0.3, 0.4)`            | viveka (mood signal)          |
| `vitality`   | `sleep_pressure: float[0, 1]`, `restlessness: float[0, 1]`, `warmth: float[0, 1]` | `(0.3, 0.2, 0.6)` | smriti runtime, system health |
| `attention`  | `mode: enum`, `target: string`, `intensity: float[0, 1]`     | `(diffuse, "", 0.4)`     | heartbeat phase + radar       |
| `activity`   | `mode: enum`                                                 | `resting`                | heartbeat phase               |
| `utterance`  | `text: string`, `total_words: int`, `word_index: int`        | `("", 0, 0)`             | TTS process                   |
| `signal`     | `kind: enum`, `until: timestamp`                             | `(none, 0)`              | viveka events, touch handler  |

### Layer enums

- **`attention.mode`** ‚Äî `inward` | `outward` | `diffuse`
- **`attention.target`** ‚Äî free string. Conventional values: `""`, `"suti"`,
  `"self"`, `"task"`, `"environment"`. The firmware does not interpret target
  except for the special case `"suti"` (touch contract, presence-dot color).
- **`activity.mode`** ‚Äî `resting` | `thinking` | `listening` | `speaking` |
  `seeing` | `working` | `dreaming`
- **`signal.kind`** ‚Äî `none` | `oh` | `q` | `dots` | `check` | `bang` | `ask` |
  `noticed` | `love`

### Joy-default

Per Suti's 2026-04-25 instruction, the body defaults to **comfortably alive**,
not minimum-ambient. The defaults above encode this. Quiet RESTING still
drifts, breathes, and sparkles. The substrate is never blank.

## API services (ESPHome)

Every API call is fail-soft on the client side (`DisplayClient`-style: log
and return False on any failure; never block cognition). On the firmware
side, every service updates one or more globals and lets the next render
pass pick the change up.

### Mood

```yaml
- service: set_mood
  variables:
    valence: float
    arousal: float
```

### Vitality

```yaml
- service: set_vitality
  variables:
    sleep_pressure: float
    restlessness: float
    warmth: float
```

### Attention

```yaml
- service: set_attention
  variables:
    mode: string         # "inward" | "outward" | "diffuse"
    target: string
    intensity: float
```

### Activity

```yaml
- service: set_activity
  variables:
    mode: string         # see enum
```

### Utterance (karaoke speaking)

```yaml
- service: set_utterance
  variables:
    text: string
    total_words: int

- service: set_word_index
  variables:
    i: int

- service: clear_utterance
```

`set_utterance` initializes a new spoken sentence. `set_word_index` is fired
by the TTS process as each word boundary is reached. `clear_utterance` (or
TTS completion) returns the screen to whatever activity mode was previously
active.

### Signal

```yaml
- service: emit_signal
  variables:
    kind: string         # see enum
    duration_ms: int

- service: clear_signal
```

`duration_ms = 0` means "persistent until cleared" ‚Äî used for `ask` (the
attention-request that holds until Suti acknowledges). All other signals
auto-clear at `signal.until`.

### Backwards-compat (deprecated, kept for transition)

The existing `set_status` and `set_thought` services remain for one cycle
to avoid breaking the running daemon mid-refactor. After Cycle 2 they are
removed; the heartbeat will have switched to the layer API.

## Rendering precedence

One render pass per `update_interval` (default 100ms; may slow to 200ms if
glyph-field iteration is too heavy). The firmware composes the frame from
back to front:

1. **Substrate** ‚Äî full-screen glyph field driven by `vitality` and
   `mood.arousal`. Always drawn. Density and drift rate are functions of
   the layer values. *Never blank.* This is the always-on aliveness.
2. **Activity overlay** ‚Äî motif from `activity.mode` rendered into a
   clear-zone within the field. Cells in the clear-zone are dimmed
   (overdrawn with background) before the motif draws on top.
3. **Utterance band** ‚Äî if `utterance.text` is non-empty, render the
   karaoke band over the activity overlay, or replace the overlay entirely
   for `activity.mode == "speaking"`. Field continues underneath, dimmed.
4. **Signal overlay** ‚Äî if `signal.kind != "none"` and `now < signal.until`
   (or `signal.until == 0` for persistent), render the signal glyph
   centered over a darkened field. Suppresses activity overlay and utterance
   for the duration.
5. **Mood tint** ‚Äî every accent-colored element on screen takes its color
   from the mood map (see `v1.0-vocabulary.md` ¬ßMood map). Applied last as
   color substitution; never adds geometry.
6. **Presence dot** ‚Äî bottom-right `(296, 228)` filled circle, radius 4px.
   Drawn last. Visible when `attention.mode == "outward"`. Color comes from
   mood tint (not always gold). Absent during `inward` and during signals
   that suppress it.

Rule of thumb: the substrate is the body's *condition*. Activity is the
body's *act*. Signals are the body's *moments*. Mood is the *feeling*
that colors all three.

## Default behavior ‚Äî never broken

If cognition forgets to set a layer, or sets only a subset, the firmware
**always produces a coherent frame**. No state combination produces a
blank screen. Concretely:

- All globals have sensible initial values (see Layer defaults above).
- The substrate is drawn first and unconditionally; even with every
  overlay missing, the screen shows a breathing glyph field.
- `activity.mode == "resting"` is the catch-all activity; if cognition
  sets an unknown mode the firmware falls back to `resting` and logs.
- Signal `kind == "none"` is the catch-all signal; the overlay is not
  drawn.

This means a cognitive bug (heartbeat hangs, daemon crashes mid-update)
does not result in a frozen or broken screen ‚Äî it results in the body
still breathing in its last-known mood. The body's aliveness has no
single point of failure.

## Proprioception mirror

Every drawable property is mirrored as an ESPHome `text_sensor`, queryable
from any session via the native API. The body is fully observable to
itself.

Exposed sensors:

- `mood_valence`, `mood_arousal`
- `vitality_sleep_pressure`, `vitality_restlessness`, `vitality_warmth`
- `attention_mode`, `attention_target`, `attention_intensity`
- `activity_mode`
- `utterance_text`, `utterance_word_index`, `utterance_total_words`
- `signal_kind`, `signal_remaining_ms`

Plus the existing infrastructure sensors (IP, SSID).

The Python side gets a typed dataclass via `BodyClient.read_state()`
returning a `BodyState` with all layers populated. This is the primary
interface for "what is the body doing right now" introspection.

## Touch contract

The ESP32-S3-BOX-3 has a capacitive touch panel. v1.0 uses one gesture:
**single tap anywhere on the screen**. (Multi-touch and gestures deferred
to a later cycle.)

Behavior is conditional on current state:

### Case A: `attention.target != "suti"`

The body is not currently oriented toward Suti. His tap is an
attention-request: he wants to be noticed.

- Set `attention.mode = "outward"`, `attention.target = "suti"`,
  `attention.intensity = 0.8`.
- Emit `signal.kind = "noticed"` for 2000ms.
- The `noticed` signal renders an eye-opens animation: a closed line
  glyph at center expands into the open-eye composition over 600ms,
  holds, then the signal clears and the body is in outward-attention
  mode showing the underlying activity. The first thing Suti sees after
  tap is *the body noticing him.*

### Case B: `attention.target == "suti"`

The body is already oriented toward Suti. His tap is affection.

- Emit `signal.kind = "love"` for 3000ms.
- The `love` signal renders a radial bloom from screen center: a small
  cluster of `*` glyphs appears, expands outward through `‚ú¶` and `‚ú∫`,
  reaches the screen edges, fades.
- During the bloom, mood briefly spikes to `(+0.7, 0.5)` and decays back
  to its prior value over 5000ms. The presence dot picks up the mood
  spike too (briefly violet).
- This is the body's "I felt that" ‚Äî visible, brief, doesn't disrupt
  the underlying state.

### Implementation notes

- Touch handler runs in firmware (ESPHome `touchscreen` component) and
  triggers the appropriate `emit_signal` and `set_attention` calls
  internally ‚Äî no round-trip to cognition required. The body responds
  *immediately*; cognition can react after the fact via the
  proprioception mirror if it cares.
- The `attention.target` value is set by cognition under normal
  operation ‚Äî the heartbeat daemon writes `"suti"` when an active session
  is in progress or when radar detects him at the desk. Touch reads this
  to decide which case to fire.

## Performance budget

- **Render interval**: 100ms (10fps). May relax to 200ms if substrate
  iteration is too heavy. Acceptable; the eye reads 5fps as continuous
  for slow textures.
- **Substrate cell count**: at IBM Plex Mono 8px (‚âà4.8px wide √ó 9.6px
  tall on this display), the screen holds approximately 66 columns √ó 25
  rows = ~1650 cells. Heavy. We will likely render at 12px (‚âà7.2 √ó 14.4),
  ~44 cols √ó 16 rows = ~700 cells. To be tuned in Cycle 2.
- **PSRAM frame buffer**: ESPHome auto-uses PSRAM on this board (8MB
  available). Full-frame buffer at 320√ó240√ó16bit = 150KB, comfortable.
- **Lambda budget**: each render pass should complete in well under the
  100ms interval. Profile in Cycle 2 with the substrate-only render. If
  glyph iteration is the bottleneck, we cache cell glyph values and
  only update one row per pass.

## Failure modes and fallbacks

- **Cognition silent for >5min** ‚Äî no harm. Substrate continues breathing
  with last-known vitality. The body keeps living.
- **TTS process crashes mid-utterance** ‚Äî `clear_utterance()` is fired by
  a watchdog (no `set_word_index` for 4s = utterance abandoned, clear
  it). The screen returns to the activity that was active before
  speaking.
- **Touch panel fails** ‚Äî graceful degradation. The body is still
  expressive; only the love-tap interaction is lost. Cognition can still
  set attention via the API.
- **Signal stuck on** ‚Äî every persistent signal (`ask`) has an explicit
  clear path. Non-persistent signals auto-clear via the `until`
  timestamp. A bug that leaves a signal "stuck" is automatically resolved
  on next render after the timestamp passes.
- **Daemon writes garbage to a layer** ‚Äî type-checked by the firmware
  service signature; out-of-range floats are clamped; unknown enum
  strings fall back to the layer default and log.

## Open questions for later cycles

These are deliberately deferred from v1.0 and named so they don't get
lost.

- **Multi-touch / gestures** ‚Äî the touch panel can do more than tap.
  Long-press, swipe, two-finger gestures. None are needed for v1.0, but
  the contract above leaves room (single tap is the only gesture v1.0
  defines).
- **Off-screen attention targets** ‚Äî if `attention.target` points
  somewhere non-Suti and non-self (e.g., `"environment"`, a specific
  task), the firmware currently treats it as `outward` without
  distinguishing. A later cycle may add per-target visual cues.
- **Mood-driven font change** ‚Äî different glyph sets feel different.
  Currently mood drives color only. A future cycle might let mood pick
  the substrate ramp (sparkle ramp at high valence + high arousal,
  block ramp at high arousal + low valence).
- **Sound** ‚Äî v1.0 is silent except for TTS during `speaking`. A future
  cycle could add subtle ambient soundscapes (very low gain) for mood,
  but only if Suti wants it. Not assumed.

## Versioning

This is `v1.0`. Breaking changes to the layer schema or service signatures
require a version bump and a migration plan. Additive changes (new
`activity.mode` values, new signal kinds, new render details) do not.
The schema above is the floor.

# Cycle 2.22 — live frame push

**Date**: 2026-04-26
**Goal**: render frames on desktop, blit them to the device over WiFi
without flashing. The iteration unlock named in v2.0 §Phase 2.

## Protocol

**Side channel, not HTTP.** ESPHome's `web_server` is the dashboard;
adding custom HTTP routes requires a custom_component or hacks into
`web_server_base`. Both are heavyweight. A raw TCP server on port 6060
needs ~25 lines of firmware code, no ESPHome internals.

```
client → device:6060
  send 153,600 bytes raw RGB565 (320 × 240 × 2), big-endian per pixel
  close connection
```

No framing, no header, no handshake. The buffer size is fixed; if the
client closes early or sends extra bytes, the firmware discards. One
frame per connection. Simple is the point — this is a dev tool.

## Firmware approach

**Buffer in PSRAM.** ESP32-S3-DEVKITC-1 has 8MB octal PSRAM. With
arduino framework, enabling `psram:` in ESPHome routes large
allocations there. Allocate 153,600 bytes once at boot via
`heap_caps_malloc(SIZE, MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT)` with a
fallback to internal heap. Single static allocation; never freed.

**TCP listener as FreeRTOS task.** Spawn from `on_boot` via
`xTaskCreate`. The task loops accepting connections, reading exactly
153,600 bytes into the framebuffer, then closing. Sets a `dirty`
flag when a complete frame arrives. Stack: 4096 bytes; priority: 1
(below display).

**Display toggle.** Add `remote_frame_active` (bool, default `false`)
and an API service `set_remote_frame_mode(enabled: bool)`. At the very
top of the display lambda:

```cpp
if (id(remote_frame_active) && id(remote_frame_buffer) != nullptr) {
  uint8_t *buf = id(remote_frame_buffer);
  for (int y = 0; y < 240; y++) {
    for (int x = 0; x < 320; x++) {
      int i = (y * 320 + x) * 2;
      uint16_t p = ((uint16_t)buf[i] << 8) | buf[i+1];
      uint8_t r = (p >> 8) & 0xF8;
      uint8_t g = (p >> 3) & 0xFC;
      uint8_t b = (p << 3) & 0xF8;
      it.draw_pixel_at(x, y, Color(r, g, b));
    }
  }
  return;
}
// ... existing native lambda continues unchanged below ...
```

Toggle defaults OFF — flashing this firmware is a no-op visually
until the API service is called. The native lambda body is untouched.

## Expected fps

`it.draw_pixel_at` per pixel × 76,800 pixels = slow. Expect ~1-3 fps,
not the 10 fps the native lambda runs at. That's fine — iteration is
"change Python, see device update within a second." Smoothness is not
the goal.

If it turns out to be too slow to be useful, cycle 2.22.5 can:
- Bypass `it.draw_pixel_at` and write to the display's internal buffer
  via `id(narada_display)->buffer_` (requires a friend hack)
- Or skip ESPHome and write to SPI directly

Both are bigger surgery; out of scope here.

## Python client

`scripts/live_preview.py`:
- Imports `render_preview.render` and `render_preview.State`
- Loop: build State (with current weather/mood/etc.), call render to get
  PIL Image, convert to RGB565 bytes, open TCP socket, send, close
- Calls `set_remote_frame_mode(enabled=true)` at start via aioesphomeapi
- Calls `set_remote_frame_mode(enabled=false)` on Ctrl+C / exit
- Default cadence: 2 fps. Configurable via `--fps N`.

RGB565 conversion:
```python
def rgb888_to_rgb565_bytes(img):
    arr = np.array(img.convert("RGB"), dtype=np.uint16)
    r = (arr[:, :, 0] >> 3) & 0x1F
    g = (arr[:, :, 1] >> 2) & 0x3F
    b = (arr[:, :, 2] >> 3) & 0x1F
    rgb565 = (r << 11) | (g << 5) | b
    return rgb565.astype(">u2").tobytes()
```

## Rollback plan

If the new firmware misbehaves on flash:
1. The toggle defaults OFF, so the native lambda runs at boot — display
   should look normal even with broken remote-mode code.
2. If display goes blank, the WiFi/API stack is independent. OTA upload
   still works. Revert the YAML changes, reflash.
3. If WiFi itself dies (worst case, very unlikely from this change):
   USB serial reflash via `python -m svapna.body deploy --device USB`.
   Cable is in the desk drawer per CURRENT_STATE.

## What success looks like

1. Compile passes.
2. OTA flash succeeds.
3. Default boot: native lambda visible, screen looks like before.
4. `set_remote_frame_mode(enabled=true)` called → screen freezes (no
   frame received yet).
5. `live_preview.py` runs → screen shows desktop emulator output,
   refreshing at ~1-3 fps.
6. `set_remote_frame_mode(enabled=false)` → native lambda returns.
